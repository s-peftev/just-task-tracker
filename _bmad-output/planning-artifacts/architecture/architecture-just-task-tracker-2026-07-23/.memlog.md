- (question) OPEN Q1 -- ACS Room lifecycle: research proposed '1 Board = 1 ACS Room' (implicitly persistent, reused). Alternative: 1 fresh Room per CallSession (created on 'start video', torn down when the session ends). Real trade-off: a persistent per-board Room is simpler now but doesn't cleanly extend to the deferred scheduled-meetings roadmap (multiple future scheduled sessions on one board can't share one Room's validFrom/validUntil window). PROPOSED RESOLUTION: one Room per CallSession (fresh each session), CallSession.AcsRoomId unique per row -- extends directly to 'pre-create a Room with a future validFrom' for scheduled meetings later, no re-architecture needed.
- (question) OPEN Q2 -- Single-presenter screen-share enforcement: ACS Calling SDK has no native 'only one presenter' concept; must be app-enforced, and only 'soft' enforcement is possible since ACS can't be told server-side to block a specific client's screen-share track. PROPOSED RESOLUTION: server-authoritative 'CurrentPresenterUserId' field on CallSession, set/cleared via a dedicated command (e.g. RequestScreenShareCommand/StopScreenShareCommand) that the client calls BEFORE invoking the ACS SDK's local start-screen-share -- closes the race between two simultaneous clicks via a DB check, follows the existing 'business rule lives in the handler' convention, and is relayed to all clients via the existing notifier/SignalR pattern so the UI enables/disables the share button live.
- (question) OPEN Q3 -- New CallsHub vs reusing BoardActionsHub: call-state events (call started/ended, presenter changed, participant joined/left) could go over a NEW dedicated CallsHub (mirroring how Boards has 2 hubs for genuinely distinct concerns: actions vs export-status background jobs) OR be added as new notification types on the EXISTING BoardActionsHub/IBoardActionNotifier, since board viewers already subscribe to the BoardActions:{boardId} group. PROPOSED RESOLUTION: reuse BoardActionsHub -- same audience (board viewers), avoids a second SignalR connection/subscription per board for the same viewers, and call-state is conceptually just another board action stream.
- (question) OPEN Q4 -- Event Grid webhook endpoint security: a new public endpoint (e.g. POST /calls/acs-events) must receive ACS Event Grid delivery, but Event Grid does not send an Entra JWT, while every other endpoint in this app requires Entra JWT auth. Real trade-off between simplicity and rigor. PROPOSED RESOLUTION: [AllowAnonymous] on this one action, protected instead by (a) validating Event Grid's subscription-validation handshake (standard requirement, rejects anything that isn't a genuine Event Grid subscription) and (b) validating the Event Grid webhook signature/token per Microsoft's secure-webhook guidance -- no AAD-app-only delivery for MVP scale, revisit if this becomes a real attack surface concern.

- (decision) Lifecycle authority: participant join/leave and session closure are driven EXCLUSIVELY by ACS Event Grid events (Microsoft.Communication.CallParticipantAdded/CallParticipantRemoved/CallEnded -- verified these exist for plain Calling SDK group calls, not just Call Automation), delivered to the existing webhook (AD-11).
